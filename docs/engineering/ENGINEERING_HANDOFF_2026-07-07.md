## Product Snapshot

Provance is building a trust infrastructure platform for synthetic media verification.

Current MVP posture:

- image-first
- waitlist-first
- invite-based access
- authenticated analyst workspace
- queue-backed scan processing
- structured in-app report review
- printable report output
- internal admin waitlist and invite operations

## Core Stack

### Frontend

- React
- Vite
- Tailwind CSS v4-style token usage through `src/index.css`
- React Router
- Framer Motion in the public site

### Backend

- NestJS in `backend/`
- versioned API prefix under `/v1`
- Supabase for auth, Postgres, and Storage
- BullMQ-compatible queue orchestration
- Upstash Redis as queue transport
- Fly.io deployment for API and worker

### Worker

- dedicated worker runtime in `backend/src/worker.ts`
- processes queued scans asynchronously

## How Auth Works

### Waitlist

The public waitlist form submits to the backend and stores records in `waitlist_applications`.

### Invite-based activation

Approved users receive an invite link generated by the admin module.

The invite link points to:

- `/accept-invite?token=...`

The backend hashes the provided token, validates the invite record, creates the Supabase user, updates the invite record, updates the waitlist status, and writes an audit event.

### Sign-in

The sign-in page calls `POST /v1/auth/sign-in`.

The backend uses Supabase Auth and returns:

- user identity
- access token
- refresh token
- session expiry
- permissions object

### Session persistence

The frontend stores session state locally and now refreshes tokens automatically through `POST /v1/auth/refresh`.

This is the current MVP approach.

### Admin authorization

Admin access currently depends on the backend environment variable:

- `ADMIN_EMAILS`

If the signed-in email appears in that allowlist, the UI exposes the admin route and the backend authorizes the request.

## How Scan Processing Works

### Step 1. Initiate scan

The frontend calls `POST /v1/scans`.

The backend:

- validates file metadata
- creates a scan record
- generates a signed upload URL to the private Supabase bucket

### Step 2. Direct upload

The browser uploads the file directly to Supabase Storage.

### Step 3. Submit scan

The frontend calls `POST /v1/scans/:scanId/submit`.

The backend:

- moves the scan into `queued`
- enqueues it into Redis if available
- falls back to inline processing only when queue configuration is absent

### Step 4. Worker execution

The worker downloads the uploaded asset and builds the current image-first evidence payload.

Current evidence extraction includes:

- SHA-256 and MD5 fingerprints
- file-header and MIME integrity checks
- metadata extraction through `exifr`
- image statistics through `jimp`
- provenance marker check for `c2pa`
- conservative MVP verdict generation

### Step 5. Report rendering

The result payload is stored on the `scans` row and rendered in:

- dashboard recent activity
- report detail view
- printable report view

## Operational Notes

- Use `backend/`, not the legacy `api/` folder, for current backend work.
- Keep `.env.example` files safe and secret-free.
- Keep real secrets in Fly.io, Vercel, and local ignored env files only.
- Use `ADMIN_EMAILS` to control admin access.
- The admin flow now depends on the applied `0003_admin_ops.sql` migration.
